The per-service reference pages document each method in isolation. This section shows how the
services **compose** into the flows you actually ship — a checkout, a returns flow, a search
experience, a marketplace storefront, a catalog sync. Each page is a self-contained, runnable
sequence using real data shapes, so you can copy a block, swap in your own ids, and go.

## How these examples work

Every snippet uses the TypeScript SDK and follows the same conventions as the rest of the docs:

- **Real shapes.** Request and response bodies mirror what the live API actually returns — real
  field names, real structure. Ids/names/emails are illustrative (`John Doe`, `example.com`) but
  the shape is exact.
- **Two clients.** Read endpoints (catalog, search, pricing) take a **publishable** key and are
  safe in the browser. Writes that touch money or another customer's data (orders, payments) take a
  **secret** key and run on your server. Each step notes which client it uses.

```typescript
import { Tybrite } from '@tybrite-labs/sdk';

// Browser / storefront — read-only, safe to ship in client code
const storefront = new Tybrite({ apiKey: 'tybrite_pk_live_YOUR_KEY' });

// Server — full read/write, kept in an environment variable, never exposed
const server = new Tybrite({ apiKey: process.env.TYBRITE_SECRET_KEY });
```

<Info>
  Not sure which key a given call needs? The
  [Key Usage by Endpoint](/authentication#key-usage-by-endpoint) table maps every endpoint to
  `pk` / `sk` / customer-session / HMAC. The walkthroughs below call it out at each step.
</Info>
