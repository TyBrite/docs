Webhooks let your systems **react to commerce events the moment they happen** — a new order, a
successful payment, a completed catalog sync — instead of polling the API on a timer. You register an
HTTPS endpoint, subscribe it to the events you care about, and Galactic Core POSTs a signed envelope
to it on every matching event.

This page follows **one endpoint through its whole life** and shows every shape the integration hits —
not just the happy path: create it (and capture the one-time secret), read its health, verify a
delivery, test it both when it **succeeds** and when it **fails**, watch it auto-disable after
repeated failures, and handle `order.paid` end to end.

All webhook management is **secret-key only** (`sk`). A publishable key returns `403`. These calls
run on your server.

```typescript
import { Tybrite } from '@tybrite-labs/sdk';

const server = new Tybrite({ apiKey: process.env.TYBRITE_SECRET_KEY });
```

## 1. Register an endpoint

Create an endpoint pointed at your handler URL and list the events to subscribe to (or `["*"]` for
all of them). The response includes a **`signing_secret` returned exactly once** — store it
immediately; it cannot be retrieved again. If you lose it, delete the endpoint and re-create it.

```typescript
const { webhook_endpoint } = await server.webhooks.createWebhookEndpoint({
  requestBody: {
    url: 'https://yourapp.com/webhooks/tybrite',
    events: ['order.created', 'order.paid', 'payment.succeeded'],
    description: 'Production ERP sync',
  },
});

// Persist webhook_endpoint.signing_secret now — you won't see it again.
```

```json Response

    "id": "ca8e7b1e-86e4-46a3-a716-eab9e5ffc168",
    "store_id": "ef7b07d3-424b-4b03-b1d4-30a2d2073b61",
    "url": "https://yourapp.com/webhooks/tybrite",
    "events": ["order.created", "order.paid", "payment.succeeded"],
    "enabled": true,

```

<Warning>
  The **create** response is the only one that ever includes `signing_secret`. `getWebhookEndpoint`

  creation. (The mirror image is also true: create has **no** health fields or `delivery_stats`;
  those only appear on get/list — see step 2.)
</Warning>

## 3. Health degradation and auto-disable

Failed deliveries are retried automatically with exponential backoff. Each failure bumps
`consecutive_failures` and stamps `last_failure_at`; a success resets the counter. If an endpoint
fails **20 deliveries in a row**, Galactic Core disables it — `enabled` flips to `false`, a
`disabled_reason` is stamped, and `disabled_at` is set. From then on it receives nothing.

A disabled endpoint reads back like this:

```typescript
const { webhook_endpoint } = await server.webhooks.getWebhookEndpoint({
  id: 'ca8e7b1e-86e4-46a3-a716-eab9e5ffc168',
});
```

```json Response
{
  "webhook_endpoint": {
    "id": "ca8e7b1e-86e4-46a3-a716-eab9e5ffc168",
    "store_id": "ef7b07d3-424b-4b03-b1d4-30a2d2073b61",
    "url": "https://yourapp.com/webhooks/tybrite",
    "events": ["order.created", "order.paid", "payment.succeeded"],
    "enabled": false,
    "metadata": {},
    "last_success_at": "2026-06-18T08:12:00.000+00:00",
    "last_failure_at": "2026-06-20T11:45:10.220+00:00",
    "consecutive_failures": 20,
    "disabled_reason": "auto_disabled_after_20_consecutive_failures",
    "disabled_at": "2026-06-20T11:45:10.220+00:00",
    "created_at": "2026-06-20T11:30:48.673+00:00",
```

Once your handler is healthy again, re-enable it by patching `enabled: true` — that also resets the
failure counter and clears `disabled_reason`:

```typescript

  id: 'ca8e7b1e-86e4-46a3-a716-eab9e5ffc168',
  requestBody: { enabled: true },
});
// → webhook_endpoint.enabled === true, consecutive_failures === 0, disabled_reason === null
```

## 4. The event envelope

Every delivery is a POST of the same envelope shape. The `data.object` holds the resource that
triggered the event; the top-level fields tell you how to handle it:

- **`type`** — the event type (e.g. `order.created`).
- **`livemode`** — `true` for a live key, `false` for a test key. Branch on this so test events never
  touch production systems.
- **`api_version`** — the contract version (`v1`).
- **`environment`** — `production` or `sandbox`.

Here's a real `order.created` envelope as it lands in your handler (lifted from the event log):

```json
{
  "id": "evt_1781955476025_lp1jafe",
  "type": "order.created",
  "api_version": "v1",
  "livemode": true,
  "created_at": "2026-06-20T11:37:56.025Z",
  "data": {
    "object": {
      "id": "e7d28e96-e9e0-45ce-be28-51812f63eb4b",
      "customer_id": "8c958f7c-918f-488a-a7cc-71497f677a53",
      "order_number": "ORD-1781955473455",
      "order_status": "pending",
      "payment_status": "paid",
      "total_amount": 603.99
    }
  }
}
```

## 5. Verify the signature in your handler

This is **your** server code, so it's free to name your own stack. Galactic Core signs every delivery
and sends the signature in the `X-Tybrite-Signature` header as `t=<timestamp>,v1=<hex hmac>`.
Recompute the HMAC over `` `${t}.${rawBody}` `` with your stored `signing_secret` and compare in
constant time **before** trusting the payload:

```typescript
// Your webhook handler (Express example) — verify before you act.
import { createHmac, timingSafeEqual } from 'crypto';
import express from 'express';

const app = express();
const SIGNING_SECRET = process.env.TYBRITE_WEBHOOK_SECRET; // the value from step 1

// Capture the RAW body — you must sign the exact bytes, not a re-serialized object.
app.post('/webhooks/tybrite', express.raw({ type: 'application/json' }), (req, res) => {
  const header = req.get('X-Tybrite-Signature') || '';
  const parts = Object.fromEntries(header.split(',').map((p) => p.split('=')));
  const rawBody = req.body.toString('utf8');

  const expected = createHmac('sha256', SIGNING_SECRET)
    .update(`${parts.t}.${rawBody}`)
    .digest('hex');

  const ok =
    parts.v1 &&
    expected.length === parts.v1.length &&
    timingSafeEqual(Buffer.from(expected), Buffer.from(parts.v1));

  if (!ok) return res.status(400).send('invalid signature');

  const event = JSON.parse(rawBody);
  // event.type, event.data.object — now safe to act on.
  res.status(200).json({ received: true });
});
```

<Tip>
  Acknowledge fast (`200` immediately) and process asynchronously. A slow handler causes timeouts,
  which count as delivery failures and push the endpoint toward auto-disable (step 3).
</Tip>

## 6. Test, audit, and retry deliveries

**Send a test event** to verify connectivity and your signature check end to end. The payload carries
`test: true` and placeholder data — your handler should no-op on it (don't fulfil or charge). The
response reports the actual `status_code` your endpoint returned.

When your handler answers `200`, the test succeeds:

```typescript
const test = await server.webhooks.sendTestWebhookEvent({
  id: 'ca8e7b1e-86e4-46a3-a716-eab9e5ffc168',
  requestBody: { event_type: 'order.created' },
});
```

```json Response
{
  "success": true,
  "event_id": "evt_test_1781955051835_uuib0s8",
  "event_type": "order.created",
  "endpoint_url": "https://yourapp.com/webhooks/tybrite",
  "status_code": 200,
  "response_body": "{\"received\":true,\"event_id\":\"evt_test_1781955051835_uuib0s8\"}"
}
```

When your endpoint is down or rejects the request, the **same call** comes back with `success: false`
and the failing `status_code` — this is what a broken endpoint looks like, and it's how a run of
these eventually trips the auto-disable in step 3:

```json Response
{
  "success": false,
  "event_id": "evt_test_1781955052910_h3p9wxa",
  "event_type": "order.created",
  "endpoint_url": "https://yourapp.com/webhooks/tybrite",
  "status_code": 500,
  "response_body": "Internal Server Error"
}
```

## 7. Handling `order.paid` end to end

Here's the important mental model: **you do not patch the order to `paid` from the webhook.** When a
payment confirms, Galactic Core already marks the order paid, reduces stock, books the sale, and
updates the customer's metrics — automatically. The `order.paid` webhook fires **so that *your* other
systems can react** — push the order into your ERP, send a custom confirmation email, update your
analytics warehouse, notify a fulfilment partner.

```typescript
// Inside your verified handler (after the signature check in step 5):
const event = JSON.parse(rawBody);

if (event.type === 'order.paid' && event.livemode) {
  const order = event.data.object;
  await pushToErp(order);              // your system of record
  await sendConfirmationEmail(order);  // your email provider
  await trackRevenue(order);           // your analytics
}
// Do NOT re-mark the order paid — Galactic Core already did, including stock + accounting.
```

## Supported event types

Subscribe to any of these (or `["*"]` for everything):

- **Order lifecycle** — `order.created`, `order.paid`, `order.fulfilled`, `order.shipped`,
  `order.cancelled`, `order.refunded`, `order.updated`
- **Payment** — `payment.succeeded`, `payment.failed`, `payment.refunded`
- **Customer** — `customer.created`, `customer.updated`, `customer.deleted`
- **Inventory & catalog** — `product.created`, `product.updated`, `product.stock_low`,
  `product.out_of_stock`
- **Cart & checkout** — `cart.created`, `cart.updated`, `cart.abandoned`
- **Gift cards** — `gift_card.issued`, `gift_card.redeemed`, `gift_card.expired`
- **Promotions** — `promotion.applied`, `promotion.created`, `promotion.activated` (a promotion went
  live), `promotion.deactivated` (a live promotion ended, was paused, or expired)
- **Content & collections** — `collection.created`, `collection.updated` (name / banner / homepage
  placement changed), `post.published`, `lookbook.published`, `review.approved`
- **Store lifecycle & configuration** — `store.updated` (name / logo / branding / contact / base
  currency; carries `changed_fields`), `payment_provider.connected`, `shipping_provider.connected`,
  `channel.connected`
- **Feature availability** — `feature.status_changed` (a capability crossed between `available`,
  `awaiting_data`, and `not_in_plan`; carries `feature`, `status`, `previous_status`)
- **Catalog sync & syndication** — `feed.sync.completed` (a scheduled inbound feed-pull finished —
  carries created/updated/failed counts), `channel.sync.completed` (a sales-channel push finished —
  carries pushed/rejected counts)
- **Wholesale (B2B)** — `b2b.rfq.created`, `b2b.quote.sent` / `.accepted` / `.rejected`,
  `b2b.po.issued` / `.confirmed` / `.fulfilled`, `b2b.invoice.issued` / `.paid` / `.overdue` (for
  stores with wholesale enabled)

## Keeping a storefront in step with the store

The order, payment, and inventory events above react to a *transaction*. The **store lifecycle**,
**content & collections**, and **feature availability** events react to the store *growing* — and they
are what an automation tool listens to when it keeps a storefront current without anyone rebuilding it.

A store rarely arrives fully-formed. A merchant starts with a thin catalog and no reviews, no
promotions, no published content — and the matching storefront surfaces have nothing to show. As the
store fills in, these events fire at the exact moment each surface earns its place:

- `promotion.activated` → render the live promotion's banner and badge the eligible products; retire
  them on `promotion.deactivated`.
- `review.approved` → show the star summary and reviews block on the product page once there is a real
  review to display.
- `collection.updated` (banner added, or `show_on_homepage` turned on) → add the collection to the
  homepage merchandising.
- `feature.status_changed` (`awaiting_data → available`) → light up a capability the moment its data
  exists — for example, switch on recommendations shelves once there is enough signal, or a currency
  switcher once a second currency is configured. A `not_in_plan → available` transition is the same
  signal after a plan upgrade unlocks the capability.

This is how **Anvil**, Galactic Core's storefront generator, powers its self-improving storefronts: it
subscribes to these events and, when one fires, proposes the matching storefront change for the
merchant to preview and approve — no rebuild, no downtime. Any tool can do the same by subscribing to
the same events.
