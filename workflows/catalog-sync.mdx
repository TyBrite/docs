title: "Catalog Sync"
description: "Push your external catalog into Galactic Core — sample a feed, dry-run validate it, ingest it under the strictest auth, and publish your catalog back out for anyone to pull."
---

Ingestion is the **one merchant-facing write surface** in the whole API. Everything else is
storefront-only; this is how *you, the merchant* push your own external catalog **into** Galactic
Core (JSON, XML, or CSV) and keep it in sync. Because it writes to your store, it's the
**strictest-authed endpoint** in the platform: the live import requires a **secret key AND an HMAC
signature** — both, every time.

The flow below is a complete sync: grab a sample feed to model yours after, **dry-run** it to catch
problems before they matter, **ingest** it for real, then publish your catalog back out so other
systems can pull it.

```typescript
import { Tybrite } from '@tybrite-labs/sdk';

// These are server-side flows — use a secret key, kept in an environment variable.
const server = new Tybrite({ apiKey: process.env.TYBRITE_SECRET_KEY });
```

<Info>
  The `/sample`, `/test`, and public-feed reads need **no key at all**. Only the live import
  (`ingestProducts`) needs `sk` + HMAC. See
  [Key Usage by Endpoint](/authentication#key-usage-by-endpoint).
</Info>

## 1. Get a sample feed to model yours after

Before you build anything, pull a real, valid sample feed. It shows the exact field names and
demonstrates both a **single-variant** product and a **multi-variant** product (variants share a
`product_group`). No key required.

```typescript
const sample = await server.ingestion.getIngestSample({ format: 'json' });
```

## 2. Dry-run validate your feed

Send your feed to the validator first. It parses every row, returns the same `summary` and `errors`
you'd get from a real import, and adds a `normalized_preview` showing exactly how Galactic Core
interpreted each row — **without writing anything**. No key required, so you can perfect the format
before you ever sign a request.

```typescript
const check = await server.ingestion.testIngest({
  requestBody: {
    products: [
      {
        sku: 'GTS-INGEST-LAMP-01',
        name: 'Aurora Pro Desk Lamp',
        category: 'Lighting',
        brand: 'Lumitech',
        price: 54.99,
        cost: 22,
        stock: 40,
        description: 'Adjustable LED desk lamp with wireless charging base.',
      },
      {
        sku: 'GTS-INGEST-MAT-01',

        brand: 'Trailhead',
        price: 44.99,
        cost: 18,
        stock: 60,
        description: 'Non-slip 6mm yoga mat.',
      },

  },
});
```

<Note>
  `status` is `valid` (no failures), `partial` (some rows rejected), or `invalid` (all rejected).
  Each entry in `errors` names the offending `row`, `sku`, `field`, and a human-readable `reason`
  (e.g. `price must be a positive number`) — fix those, re-run, and only sign the live request once
  you see `status: valid`.
</Note>

## 3. Push the catalog (secret key + HMAC)

This is the real write. It is **upsert-by-SKU**: a `sku` that already exists is **updated**, a new
one is **created**. Rows are applied independently — one bad row never fails the batch — and the
response carries created / updated / skipped / failed counts plus an `errors` array for any
rejections.

It requires three signing headers, the **same scheme as orders and payments**: an
`Idempotency-Key` (retrying with the same key returns the original result instead of re-importing),
an `X-Timestamp` (unix seconds, within 5 minutes of server time), and an `X-Signature` — the base64
HMAC-SHA256 of `` `${timestamp}.${body}` `` signed with your store's signing secret (Settings →
Integration Settings).

```typescript
import { createHmac } from 'crypto';

const payload = {
  products: [
    {
      sku: 'GTS-INGEST-LAMP-01',
      name: 'Aurora Pro Desk Lamp',
      category: 'Lighting',
      brand: 'Lumitech',
      price: 54.99,
      cost: 22,
      stock: 40,
      description: 'Adjustable LED desk lamp with wireless charging base.',
    },
    {
      sku: 'GTS-INGEST-MAT-01',
      name: 'Studio Yoga Mat',
      category: 'Sports Equipment',
      brand: 'Trailhead',
      price: 44.99,
      cost: 18,
      stock: 60,
      description: 'Non-slip 6mm yoga mat.',
    },
  ],
};

const timestamp = Math.floor(Date.now() / 1000);
const body = JSON.stringify(payload);
const signature = createHmac('sha256', process.env.TYBRITE_HMAC_SECRET)
  .update(`${timestamp}.${body}`)
  .digest('base64');

const result = await server.ingestion.ingestProducts({
  idempotencyKey: `ingest-${timestamp}`,
  xTimestamp: timestamp,
  xSignature: signature,
  requestBody: payload,
});
```

```json Response
{
  "status": "success",
  "dry_run": false,
  "format": "json",
  "strategy": "upsert_by_sku",
  "summary": {
    "rows_total": 2,
    "rows_created": 2,
    "rows_updated": 0,
    "rows_skipped": 0,
    "rows_failed": 0
  },
  "errors": []
}
```

<Warning>
  This is the only place in the API where `sk` **and** HMAC are both mandatory. A publishable key
  returns `403`; a missing or invalid signature returns `401` (`invalid_signature`). If **every** row
  fails validation, nothing is written and the status is `422`.
</Warning>

A few rules worth knowing before your first import:

- **Category must already exist.** Categories are a fixed platform-level set — a row whose `category`
  doesn't match one of your enabled categories is rejected (it is **never** auto-created). Match a
  real category name exactly.
- **Subcategory is auto-created.** Subcategories are yours to define freely; an unknown one is
  created on the fly. Nest with ` > ` (e.g. `category: "Electronics"`, `subcategory: "Audio > Headphones"`).
- **Variants group by `product_group`.** Rows sharing a `product_group` become variants of one
  product; each still needs its own unique `sku`.
- **CSV / XML work identically.** Pass `format: 'csv'` (or `'xml'`) and a string body; the upsert and
  partial-success rules are the same.

## 4. Publish your catalog back out

The inverse direction: opt in (admin → Catalog Sync → **"Publish my catalog"**) and your catalog
becomes readable at a stable public URL — **no key required** — so any system can pull it: another
store's scheduled sync, a partner integration, a script. Only storefront-safe fields are exposed,
never cost or margin.

```typescript
const feed = await server.ingestion.getStoreCatalogFeedJson({
  store: 'ef7b07d3-424b-4b03-b1d4-30a2d2073b61', // store id or short store code
});
```
