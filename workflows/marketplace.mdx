A **marketplace** is a Galactic Core deployment that aggregates the catalogs of many independent
merchants into one storefront — think a curated multi-vendor shop where a single shopper basket can
span several sellers. This page follows **one continuous operator + shopper scenario** and shows
every shape an operator integration actually hits — not just the happy path: discovering the combined
catalog (and narrowing it to a single merchant), checking out a cart that crosses **two** merchants in
one payment, watching that one payment split into **per-merchant child orders** with commission taken
out, reading the shopper's unified cross-merchant profile, and rendering sponsored placements.

We carry two real merchants through the whole scenario:

- **Galactic Core Store** — `086e8d7f-b84d-41bc-8acd-3f5740ea61dc` (we call it merchant **A**)
- **Galactic Test Store** — `ef7b07d3-424b-4b03-b1d4-30a2d2073b61` (merchant **B**)

## Operator keys vs. store keys

There are two kinds of API key, and the difference drives everything on this page:

- A **store key** (`tybrite_pk_*` / `tybrite_sk_*`) is scoped to **one merchant**. Catalog reads
  return only that store's products, and writes act on that store alone. This is what a single
  merchant's own storefront uses.
- An **operator key** is issued for a **marketplace deployment** and aggregates across **all active
  merchants** in that marketplace. Discovery reads return the combined catalog (each row stamped
  with its source `merchant_store_id`), and the operator can run a unified checkout that fans a
  single basket out to the right merchants.

The operator key comes in both flavours too. Use the **publishable** operator key
(`tybrite_pk_*`) for the read surfaces a storefront calls from the browser — discovery, marketplace
info, collections, ad slots — and the **secret** operator key (`tybrite_sk_*`) on your server for the
unified checkout (it starts a payment) and the unified shopper profile.

```typescript
import { Tybrite } from '@tybrite-labs/sdk';

// Browser-safe operator client — discovery, info, collections, ad slots.
const storefront = new Tybrite({ apiKey: 'tybrite_pk_live_OPERATOR_KEY' });

// Server-side operator client — unified checkout + unified shopper profile.
const server = new Tybrite({ apiKey: 'tybrite_sk_live_OPERATOR_KEY' });
```

<Info>
  Operator keys work on the **discovery and marketplace** surfaces (products, categories, search,
  reviews, marketplace info, unified checkout, the unified profile, ad slots, collections, and
  cross-merchant trending recommendations). Per-merchant surfaces like promotions, gift cards, posts,
  and per-merchant cart writes reject an operator key with `403` — those belong to a single
  merchant's own store key.
</Info>

## SDK calls in marketplace mode — a quick map

Most of the **standard GC SDK calls work unchanged** under an operator key — you build a marketplace
storefront with the same `client.products` / `client.search` / `client.cartWishlist` / `client.returns`
you already know. The table is what changes, and where a marketplace-only call replaces a standard one.
Read this before the flows below.

| SDK call | In marketplace mode | What changes |
| --- | --- | --- |
| `client.products.*`, `client.taxonomy.*`, `client.search.*` | ✅ same call | Returns the **aggregated** catalog across all merchants (each row carries `merchant_store_id`). Pass `storeId` to narrow to one merchant's shop page. |
| `client.recommendations.getRecommendations` | ✅ same call | Cross-merchant rail; each item stamped with its `merchant_store_id`. |
| `client.cartWishlist.*` | ✅ same call | Per-shopper cart/wishlist as usual (items carry `merchant_store_id`). |
| `client.reviews.listReviews` | ⚠️ `storeId` required | An operator key must pass the merchant `storeId` (the product's owner). |
| `client.returns.*` (createReturn / listReturns / getReturn / getStoreCredit / acceptReturnCredit / requestReturnRefund) | ⚠️ `storeId` required | A return belongs to **one merchant** — pass that merchant's `storeId` + the shopper's `xAuthToken`. There is **no** cross-merchant returns aggregate. |
| `client.promotions.*` | ⚠️ `storeId` required, read/calc only | An operator key can **read/calculate** one promotion (`getPromotion` / `calculatePromotionDiscount` / `calculateBestPromotion`) with the merchant `storeId`, but **cannot list** promotions. |
| `client.system.getStoreInfo` | ❌ use marketplace instead | Returns `403` for an operator key. Use `client.marketplace.getMarketplaceInfo()` (marketplace identity) or `getMarketplaceInfo({ storeId })` (one merchant's store info — same shape). |
| Single-merchant `client.orders.createOrder` | ❌ use marketplace checkout | A marketplace basket spans merchants → use `client.marketplace.marketplaceCheckout(...)` (one payment, auto-split). Don't call `createOrder` per merchant. |
| `client.giftCards.*`, `client.cms.*`, `client.customers.getCustomer` | ❌ per-merchant only | These are single-store surfaces — an operator key gets `403`. |
| **Marketplace-only** (no standard equivalent) | `marketplaceCheckout`, `getMarketplaceCustomer` (unified profile, + `X-Customer-Token`), `getMarketplaceInfo`, `listMarketplaceCollections` / `getMarketplaceCollection`, `getAdSlot` / `logAdEvent` | These exist **only** in marketplace mode under an operator key. |

The rule of thumb: **discovery + cart + recommendations are aggregated automatically; anything tied to one merchant (returns, reviews, promotions, store info) takes that merchant's `storeId`; and checkout / unified profile / collections / ads are marketplace-only.**

## What's storefront API vs. operator back-office

<Warning>
  This page documents the **storefront / shopper** half of a marketplace — the calls a storefront
  makes to render and transact. The **operator back-office** half — everything that *configures* the
  marketplace — is **not** in the API. Do not emit API calls for it.

  - **Storefront API (this page):** aggregated discovery, marketplace info, unified split checkout,
    reading the unified shopper profile, reading curated collections, filling ad slots, firing ad
    impression/click beacons, and per-merchant returns.
  - **Operator / merchant back-office (NOT the API — done in admin):** curating collections, ad slot
    / rate-card / booking management, setting commission rules, running payout batches, merchant
    onboarding & application approval, and a merchant's Stripe Connect payout onboarding. There is
    no public endpoint for any of these — they live in the operator's and merchants' admin.

  The tell: if it *configures* the marketplace (who can sell, what commission, which ad runs, when a
  payout pays), it's back-office. If it *renders or transacts the storefront*, it's the API.
</Warning>

## 1. Aggregated discovery — combined, then narrowed

merchant. Each product row carries a `merchant_store_id` identifying the merchant that sells it — so
the **same** `listProducts` call returns rows from **multiple** merchants.

## 2. Marketplace info

Read the marketplace's own identity and branding to render the storefront shell — name, logo,
tagline, contact, and whether unified checkout and commission are enabled. Operator key required; a
single-store key gets `403` (it should call `getStoreInfo` instead).

```typescript
const info = await storefront.marketplace.getMarketplaceInfo({});
```

```json Response
{
  "marketplace": true,
  "mode": "marketplace",
  "name": null,
  "logo_url": null,
  "primary_color": null,
  "tagline": null,
  "support_email": null,
  "description": null,
  "website": null,
  "phone": null,
  "address": null,
  "unified_checkout": true,
  "commission_enabled": true
}
```

The `marketplace`, `mode`, `unified_checkout`, and `commission_enabled` flags always come back; the
branding fields (`name`, `logo_url`, `primary_color`, `tagline`, contact, address) are `null` until
the operator sets them in their admin — render sensible fallbacks when they're absent.

<Note>
  Pass `storeId` to get **one merchant's** full store info (same shape as a single-store
  `getStoreInfo`) — useful for a per-merchant "shop page":
  `storefront.marketplace.getMarketplaceInfo({ storeId: 'ef7b07d3-424b-4b03-b1d4-30a2d2073b61' })`.
  The merchant must be active in the marketplace, otherwise `404`.
</Note>

<Note>
  **Who owns currency vs. pricing in a marketplace.** The storefront **currency is marketplace-wide**
  — it's the operator's deployment-level default, so the whole aggregated catalog renders in one
  currency (you don't get a different currency per merchant). **Dynamic pricing rules, however, are
  per-merchant** — each merchant authors their own pricing rules for their own products, and the
  operator can't set a merchant's prices. So: currency is the operator's; product prices (and any
  price rules behind them) are each merchant's. Commission/payout rules are the operator's
  (configured in their admin), separate from both.
</Note>

## 3. Unified split checkout — a cart that spans two merchants

The headline capability: a single shopper cart that spans **multiple merchants** checks out as one
order with **one payment**. We carry both merchants from step 1 forward into a single basket — items
from merchant **A** (Galactic Core Store) and merchant **B** (Galactic Test Store). Each line item
names the merchant it belongs to (`merchant_store_id`); the marketplace validates the items, reserves
stock per merchant, resolves commission **per merchant**, and starts **one** payment for the whole
basket. This is a server-side call — use the **secret** operator client.

```typescript
const checkout = await server.marketplace.marketplaceCheckout({
  requestBody: {
    items: [
      {
        product_id: 'bb63bc9c-e39b-429a-a07b-0648a8edd677',
        variant_id: 'c7d6321b-42b4-4ce5-9819-915e2ae26c7f',
        merchant_store_id: '086e8d7f-b84d-41bc-8acd-3f5740ea61dc', // merchant A
        quantity: 1,

        product_id: 'de4aeb12-6621-4a3b-b83f-6ca493aeae6e',
        variant_id: 'b0302f76-3e39-4744-8401-33ef80c25a6a',
        merchant_store_id: 'ef7b07d3-424b-4b03-b1d4-30a2d2073b61', // merchant B
```

`merchant_breakdown` has **one entry per merchant** in the basket. There is still exactly **one**
`client_secret`: the shopper pays `total_amount` once, on the storefront, and the marketplace takes
care of splitting the money.

### The commission math, concretely

The operator earns a commission on each merchant's slice (here ~8%, set by the
`commission_rule_id` the operator configured in their admin). Read it straight off the breakdown:

| Merchant | `gross_amount` | `commission_amount` | `net_amount` |
| --- | --- | --- | --- |
| A — Galactic Core Store | $415.80 | $33.26 (8%) | $382.54 |
| B — Galactic Test Store | $149.00 | $11.92 (8%) | $137.08 |
| **Group total** | **$564.80** | **$45.18** | **$519.62** |

For each merchant, `gross_amount − commission_amount = net_amount` (415.80 − 33.26 = 382.54), and the
group's `total_amount` is the sum of the gross amounts — what the shopper pays in one payment. The
operator keeps the summed commission; each merchant is owed their summed net.

<Note>
  Stock is **reserved** at checkout (each merchant slice gets its own `reservation_ids`) so concurrent
  shoppers can't oversell the last units; if an item can't be held, the call returns
  `400 insufficient_stock` and **no order or payment is created**. Marketplace-wide operator
  promotions apply automatically and are not passed. If the payment provider can't be reached the
  call returns `502 provider_error`.
</Note>

### Applying a merchant's own discount to its slice

Pass the optional `discounts` array to reduce **only one merchant's** subtotal with that merchant's
own promotion and/or gift card. The response then carries a non-empty `discount_breakdown`, an
`operator_funded_discount` total, and per-merchant `discount_amount` — operator-funded promotions are
attributed back to the operator, so you can see who paid for the discount:

```typescript
const checkout = await server.marketplace.marketplaceCheckout({
  requestBody: {
    items: [
      {
        product_id: 'bb63bc9c-e39b-429a-a07b-0648a8edd677',
        variant_id: 'c7d6321b-42b4-4ce5-9819-915e2ae26c7f',
        merchant_store_id: '086e8d7f-b84d-41bc-8acd-3f5740ea61dc', // merchant A
        quantity: 2,
      },
    ],
    customer_email: 'john.doe@example.com',
    customer_name: 'John Doe',
    currency: 'USD',
    discounts: [
      {
        merchant_store_id: '086e8d7f-b84d-41bc-8acd-3f5740ea61dc',
        promotion_id: '8a7b6c5d-1234-5678-9abc-def012345678',
        gift_card_code: 'GC-XXXX-YYYY',
      },
    ],
  },
});
```

The `discounts` array is keyed by `merchant_store_id`; each entry's `promotion_id` and/or
`gift_card_code` apply to **that merchant's** portion only. In the response, `discount_breakdown`
gains one entry per discounted merchant (`merchant_discount`, `gift_card_amount`, and the
`operator_funded_discount` share), `discount_total` is the basket-wide reduction, and that merchant's
`merchant_breakdown` entry shows its `discount_amount` and reduced `gross_amount`.

<Note>
  **The server is the price authority — you send identifiers, never amounts.** Just like the
  single-store checkout, the multi-merchant checkout recomputes every item price from the catalog and
  resolves discounts itself: a `discounts` entry only *names* a `promotion_id` / `gift_card_code`, and
  the server independently computes the real discount that promotion or gift card grants (plus the
  operator-funded share), every `gross_amount` / `discount_amount` / `merchant_gross` / commission, and
  the one payment total — then charges exactly that. A named discount that doesn't validate (expired,
  ineligible, wrong merchant) returns `400 discount_invalid` rather than being silently applied. There is
  no client-supplied `discount_amount` or per-item price in the request, so a manipulated basket can't
  lower what's charged or inflate a merchant's payout. The resolved breakdown is held server-side and
  used verbatim to finalize the per-merchant child orders after payment succeeds — nothing sent after
  checkout can change it.
</Note>

## 4. Fulfillment → per-merchant child orders

The shopper completes the single payment with the `client_secret` from step 3. **You do not call a
fulfill endpoint** — fulfillment is driven internally off the payment-success event. The observable
outcome is what you build against:

- The paid order group splits into **one child order per merchant** — each lands in that merchant's
  own order list (with its `merchant_net_amount` and `commission_amount`), so each merchant fulfills
  and ships only their own items.
- The operator's split ledger records the gross / commission / net per merchant (the same numbers
  from the `merchant_breakdown`), and each merchant's net is queued for payout to their connected
  Stripe account.
- The group flips to `paid`.

The internal fulfillment step produces this shape (shown so you know exactly what landed — the
storefront never calls it):

```json
{
  "message": "Marketplace order fulfilled",
  "order_group_id": "6b636961-9d7d-419e-b0ea-a74daa707273",
  "children": [
    {
      "store_id": "086e8d7f-b84d-41bc-8acd-3f5740ea61dc",
      "online_order_id": "ed385518-9585-4ab8-9b77-e7b8f61f2dde",
      "net_amount": 382.54
    },
    {
      "store_id": "ef7b07d3-424b-4b03-b1d4-30a2d2073b61",
      "online_order_id": "44afa54a-9509-4cf7-a75f-e9c2a47dcdbb",
      "net_amount": 137.08
    }
  ],
  "warnings": [
    "transfer (086e8d7f-b84d-41bc-8acd-3f5740ea61dc): Your destination account needs to have at least one of the following capabilities enabled: transfers, crypto_transfers, or legacy_payments.",
    "transfer (ef7b07d3-424b-4b03-b1d4-30a2d2073b61): merchant has no connected payout account"
  ]
}
```

Each `children` entry maps a merchant to the `online_order_id` created in **that merchant's** order
list, carrying the same `net_amount` from the checkout breakdown ($382.54 for A, $137.08 for B — the
two nets that summed to $519.62 above).

<Note>
  **Payout depends on a connected merchant account.** The `warnings` array reports the real Stripe
  Connect transfer state per merchant. A merchant who hasn't finished **Stripe Connect payout
  onboarding** (done in the merchant's admin, not via the API) has **no connected payout account**,
  so the transfer of their net can't be made yet — their child order is still created and the shopper
  is still charged, but their funds wait until they connect a payout account. An empty `warnings`
  array means every merchant's transfer went through.
</Note>

The same shopper may have a distinct `customer` record at each merchant — `customer.ids` lists all
of them. The `stores` array is the shopper's per-merchant relationship (with `preferred_store` and
spend totals), and `orders` is their order history across every merchant, newest first.

<Note>
  **Returns are per-merchant.** A return always belongs to the merchant whose order it is — there's
  no marketplace-wide return. A shopper lodges and tracks a return against the specific merchant
  order, so the return call **requires that merchant's `storeId`**, and that merchant approves and
  processes it in their own admin. See [Returning customer](/workflows/returning-customer) for the
  returns flow.
</Note>

## 6. Sponsored placements and curated sections

The marketplace operator monetizes the storefront with **sponsored ad slots** and merchandises it
with **curated collections**. Both are operator-key, read-only on the render side. The *bookings,
rate cards, and curation* behind them are operator back-office (see the boundary callout above) —
the storefront only **reads** the resolved placements and **fires beacons**.

### Fill an ad slot

Ask for the active placements in a named slot — a hero banner, a category strip, search results.
A slot can return **sponsored** placements (`sponsored: true`, paid — render with `disclosure_label`)
and **curated fallback** placements (operator hand-picks, not labelled). Pass `context` to scope to a
category or search term.

```typescript
const slot = await storefront.marketplace.getAdSlot({
  slotKey: 'home_hero',
  limit: 5,
});
```

```json Response
{
  "slot": "home_hero",
  "context": null,
  "placements": []
}
```

An empty `placements` array means there's no active booking (sponsored or curated) for the slot right
now — fall back to your own organic content. When a placement is live it looks like this:

```json
{
  "slot": "home_hero",
  "context": null,
  "placements": [
    {
      "booking_id": "7c9e6679-7425-40de-944b-e07fc1f90ae7",
      "merchant_store_id": "086e8d7f-b84d-41bc-8acd-3f5740ea61dc",
      "placement_kind": "product",
      "product_ids": [
        "bb63bc9c-e39b-429a-a07b-0648a8edd677"
      ],
      "creative": {
        "headline": "Summer sale",
        "image_url": "https://cdn.example.com/ads/hero.png",
        "cta": "Shop now"
      },
      "sponsored": true,
      "disclosure_label": "Sponsored"
    }
  ]
}
```

### Log impression and click beacons

After rendering, fire a beacon: one `impression` per placement when it becomes visible, and a
`click` when the shopper clicks it. Pass the placement's `booking_id`. The response is `202 Accepted`
with `billable` telling you whether the event counted toward billing after de-duplication — purely
informational; the storefront doesn't act on it.

```typescript
// When the placement scrolls into view:
const impression = await storefront.marketplace.logAdEvent({
  requestBody: {
    booking_id: '7c9e6679-7425-40de-944b-e07fc1f90ae7',
    event_type: 'impression',
    session_id: 'sess_abc123',
  },
});
// → { recorded: true, billable: true }

// When the shopper clicks it:
const click = await storefront.marketplace.logAdEvent({
  requestBody: {
    booking_id: '7c9e6679-7425-40de-944b-e07fc1f90ae7',
    event_type: 'click',
    product_id: 'bb63bc9c-e39b-429a-a07b-0648a8edd677',
    session_id: 'sess_abc123',
  },
});
// → { recorded: true, billable: true }
```

<Note>
  `billable` can come back `false` (still `recorded: true`) when the beacon was de-duplicated or
  filtered as invalid traffic — a repeat impression in the same session, for example. Treat the field
  as informational; never block rendering on it.
</Note>

### Curated merchandising collections

List the operator's curated sections ("Trending now", "Featured shops"), then resolve one section's
members. Curation itself is operator back-office; the storefront only reads the result.

```typescript
const { collections } = await storefront.marketplace.listMarketplaceCollections({});
```

```json Response
{
  "collections": []
}
```

An empty `collections` array means the operator hasn't curated any sections yet. When sections exist,
each carries its `slug`, `name`, `placement_kind`, and homepage flags:

```json
{
  "collections": [
    {
      "slug": "trending-now",
      "name": "Trending now",
      "description": "The most popular products across the marketplace this week.",
      "placement_kind": "product",
      "display_priority": 10,
      "show_on_homepage": true
    },
    {
      "slug": "featured-shops",
      "name": "Featured shops",
      "description": "Merchants worth a look.",
      "placement_kind": "merchant",
      "display_priority": 20,
      "show_on_homepage": true
    }
  ]
}
```

Resolve one section's members with its slug:

```typescript
const section = await storefront.marketplace.getMarketplaceCollection({
  slug: 'trending-now',
  limit: 12,
});
```

```json Response
{
  "slug": "trending-now",
  "name": "Trending now",
  "description": "The most popular products across the marketplace this week.",
  "placement_kind": "product",
  "curated": true,
  "items": [
    {
      "merchant_store_id": "086e8d7f-b84d-41bc-8acd-3f5740ea61dc",
      "product_id": "bb63bc9c-e39b-429a-a07b-0648a8edd677"
    },
    {
      "merchant_store_id": "ef7b07d3-424b-4b03-b1d4-30a2d2073b61",
      "product_id": "0186665f-f0a1-4c79-976a-919b5b1c2daa"
    }
  ]
}
```

<Tip>
  Filter the section list by kind with `listMarketplaceCollections({ placementKind: 'merchant' })`.
  An unknown slug on `getMarketplaceCollection` returns `404`.
</Tip>
