A **marketplace** is a Galactic Core deployment that aggregates the catalogs of many independent
merchants into one storefront — think a curated multi-vendor shop where a single shopper basket can
span several sellers. This page follows **one continuous operator scenario** and shows every shape an
operator integration actually hits — not just the happy path: discovering the combined catalog (and
narrowing it to a single merchant), checking out a cart that crosses **two** merchants in one
payment, reading a shopper's unified profile, and rendering sponsored placements.

## Operator keys vs. store keys

There are two kinds of API key, and the difference drives everything on this page:

- A **store key** (`tybrite_pk_*` / `tybrite_sk_*`) is scoped to **one merchant**. Catalog reads
  return only that store's products, and writes act on that store alone. This is what a single
  merchant's own storefront uses.
- An **operator key** is issued for a **marketplace deployment** and aggregates across **all active
  merchants** in that marketplace. Discovery reads return the combined catalog (each row stamped
  with its source `merchant_store_id`), and the operator can run a unified checkout that fans a
  single basket out to the right merchants.

```typescript
import { Tybrite } from '@tybrite-labs/sdk';

// The marketplace operator client — aggregates across all merchants in the marketplace.
const client = new Tybrite({ apiKey: 'tybrite_pk_live_OPERATOR_KEY' });
```

<Info>
  Operator keys work on the **discovery and marketplace** surfaces (products, categories, search,
  reviews, marketplace info, unified checkout, the unified profile, ad slots, and collections).
  Per-merchant surfaces like promotions, gift cards, posts, recommendations, and per-merchant cart
  writes reject an operator key with `403` — those belong to a single merchant's own store key.
</Info>

## 1. Aggregated discovery — combined, then narrowed

Discovery endpoints called with an **operator key** return results aggregated across every active
merchant. Each product row carries a `merchant_store_id` identifying the merchant that sells it — so
the **same** `listProducts` call returns rows from **multiple** merchants.

```typescript
const { products } = await client.products.listProducts({ limit: 20 });
```

```json Response
{
  "products": [
    {
      "id": "0186665f-f0a1-4c79-976a-919b5b1c2daa",
      "variant_id": "e98631a9-7852-4fe8-8efe-988935d5922a",
      "name": "4K Action Camera",
      "price": 149,
      "sale_price": 129,
      "merchant_store_id": "ef7b07d3-424b-4b03-b1d4-30a2d2073b61"
    },
    {
      "id": "715b8c10-eb11-4289-8c0e-2966e78cf9a5",
      "variant_id": "9cd6e472-de84-435c-be43-957b56682f43",
      "name": "Trail Bottle 750ml",
      "price": 24.99,
      "sale_price": null,
      "merchant_store_id": "a91c33e7-2f04-4b6a-8c19-77d2b5e90c41"

```

Two merchants in one page: the camera is sold by `ef7b07d3-…` (Galactic Test Store), the bottle by
`a91c33e7-…` (Northwind Outfitters). To scope discovery to **one** merchant within the marketplace,
narrow with `storeId` — the same call, now returning only that merchant's rows:

```typescript
const { products } = await client.products.listProducts({
  storeId: 'ef7b07d3-424b-4b03-b1d4-30a2d2073b61',
  limit: 20,
});
```

```json Response
{
  "products": [
    {
      "id": "0186665f-f0a1-4c79-976a-919b5b1c2daa",
      "variant_id": "e98631a9-7852-4fe8-8efe-988935d5922a",
      "name": "4K Action Camera",
      "price": 149,
      "sale_price": 129,
      "merchant_store_id": "ef7b07d3-424b-4b03-b1d4-30a2d2073b61"
    }
  ],
  "pagination": { "limit": 20, "next_cursor": null, "has_more": false }
}
```

<Tip>
  `client.taxonomy.listCategories()` returns a deduped union of categories across merchants, and
  `client.search.searchProducts({ q: 'wireless headphones' })` returns cross-merchant results — all
  read-only, all operator-key. Reviews also work cross-merchant: `client.reviews.listReviews({
  productId, storeId })` reads a merchant's approved reviews (for an operator key, `storeId` — the
  merchant who owns the product — is required).
</Tip>

## 2. Marketplace info

Read the marketplace's own identity and branding to render the storefront shell — name, logo,
tagline, contact, and whether unified checkout and commission are enabled. Operator key required; a
single-store key gets `403` (it should call `getStoreInfo` instead).

```typescript
const info = await client.marketplace.getMarketplaceInfo({});
```

```json Response
{
  "marketplace": true,
  "mode": "marketplace",
  "name": "Galactic Market",
  "logo_url": "https://cdn.example.com/marketplace-logo.png",
  "primary_color": "#3B82F6",
  "tagline": "Everything you need, from every merchant",
  "support_email": "support@galacticmarket.example.com",
  "description": "A curated marketplace of independent merchants.",
  "website": "https://galacticmarket.example.com",
  "phone": "+1-555-0100",
  "address": "123 Market Street, San Francisco, CA",
  "unified_checkout": true,
  "commission_enabled": true
}
```

<Note>
  Pass `storeId` to get **one merchant's** full store info (same shape as a single-store
  `getStoreInfo`) — useful for a per-merchant "shop page":
  `client.marketplace.getMarketplaceInfo({ storeId: 'ef7b07d3-424b-4b03-b1d4-30a2d2073b61' })`. The
  merchant must be active in the marketplace, otherwise `404`.
</Note>

## 3. Unified split checkout — a cart that spans two merchants

The headline capability: a single shopper cart that spans **multiple merchants** checks out as one
order with **one payment**. We carry the two merchants from step 1 forward into a single basket —
the camera from `ef7b07d3-…` and the bottle from `a91c33e7-…`. Each line item names the merchant it
belongs to (`merchant_store_id`); the marketplace validates the items, reserves stock, resolves
commission **per merchant**, and starts a single payment for the whole basket.

```typescript
const checkout = await client.marketplace.marketplaceCheckout({
  requestBody: {
    items: [
      {
        product_id: '0186665f-f0a1-4c79-976a-919b5b1c2daa',
        variant_id: 'e98631a9-7852-4fe8-8efe-988935d5922a',
        merchant_store_id: 'ef7b07d3-424b-4b03-b1d4-30a2d2073b61', // camera merchant
        quantity: 1,
      },
      {
        product_id: '715b8c10-eb11-4289-8c0e-2966e78cf9a5',
        variant_id: '9cd6e472-de84-435c-be43-957b56682f43',
        merchant_store_id: 'a91c33e7-2f04-4b6a-8c19-77d2b5e90c41', // bottle merchant
        quantity: 1,
      },
    ],
    customer_email: 'john.doe@example.com',
    customer_name: 'John Doe',
    currency: 'USD',
    shipping_address: { line1: '350 Fifth Avenue', city: 'New York', country: 'US' },
    billing_address: { line1: '350 Fifth Avenue', city: 'New York', country: 'US' },
  },
});
// One client_secret covers BOTH merchants — complete the single payment on the storefront.
```


`merchant_breakdown` has **one entry per merchant** in the basket — gross, discount, commission, and
net the marketplace resolved for each. There is still exactly **one** `client_secret`. **Once payment
succeeds the order finalizes automatically**: it fans out into one child order per merchant, each
merchant's net payout is split out, and stock is reduced. You don't make a second call.

### Applying a merchant's own discount to its slice

Pass the optional `discounts` array to reduce **only one merchant's** subtotal. The response then
carries a non-empty `discount_breakdown`, an `operator_funded_discount` total, and per-merchant
`discount_amount` — the shape the schema example demonstrates:

```typescript

  requestBody: {
    items: [
      {
        product_id: '11111111-1111-1111-1111-111111111111',
        variant_id: '22222222-2222-2222-2222-222222222222',
        merchant_store_id: '3f1a2b4c-1111-2222-3333-444455556666',
        quantity: 2,
      },
    ],
    customer_email: 'john.doe@example.com',
    customer_name: 'John Doe',
    currency: 'USD',
```

  provider can't be reached the call returns `502 provider_error`.
</Note>

identity, the merchants they have a relationship with, and their full cross-merchant order history —
in one call. Sign the shopper in first, then pass that session token in `xCustomerToken`.

```typescript
const profile = await client.marketplace.getMarketplaceCustomer({
```

## 5. Sponsored placements and curated sections

The marketplace operator monetizes the storefront with **sponsored ad slots** and merchandises it
with **curated collections**. Both are operator-key, read-only on the render side.

### Fill an ad slot

Ask for the active placements in a named slot — a hero banner, a category strip, search results.
A slot can return **sponsored** placements (`sponsored: true`, paid — render with `disclosure_label`)
and **curated fallback** placements (operator hand-picks, not labelled). Pass `context` to scope to a
category or search term.

```typescript
const slot = await client.marketplace.getAdSlot({
  slotKey: 'home_hero',
  limit: 5,
});
```

```json Response
{
  "slot": "home_hero",
  "context": null,
  "placements": [
    {
      "booking_id": "7c9e6679-7425-40de-944b-e07fc1f90ae7",
      "merchant_store_id": "3f1a2b4c-1111-2222-3333-444455556666",
      "placement_kind": "product",
      "product_ids": [
        "715b8c10-eb11-4289-8c0e-2966e78cf9a5",
        "9cd6e472-de84-435c-be43-957b56682f43"
      ],
      "creative": {
        "headline": "Summer audio sale",

        "cta": "Shop now"
      },

      "disclosure_label": "Sponsored"

```

### Log impression and click beacons

After rendering, fire a beacon: one `impression` per placement when it becomes visible, and a
`click` when the shopper clicks it. Pass the placement's `booking_id`. The response is `202 Accepted`
with `billable` telling you whether the event counted toward billing after de-duplication — purely
informational; the storefront doesn't act on it.

```typescript
// When the placement scrolls into view:
const impression = await client.marketplace.logAdEvent({
  requestBody: {
    booking_id: '7c9e6679-7425-40de-944b-e07fc1f90ae7',
    event_type: 'impression',
    session_id: 'sess_abc123',
  },
});
// → { recorded: true, billable: true }

// When the shopper clicks it:
const click = await client.marketplace.logAdEvent({
  requestBody: {
    booking_id: '7c9e6679-7425-40de-944b-e07fc1f90ae7',
    event_type: 'click',
    product_id: '715b8c10-eb11-4289-8c0e-2966e78cf9a5',
    session_id: 'sess_abc123',
  },
});
// → { recorded: true, billable: true }
```

```json Response
{ "recorded": true, "billable": true }
```

<Note>
  `billable` can come back `false` (still `recorded: true`) when the beacon was de-duplicated or
  filtered as invalid traffic — a repeat impression in the same session, for example. Treat the field
  as informational; never block rendering on it.
</Note>
