title: "Reviews & Messaging"
description: "Post-purchase engagement — submit and vote on product reviews, then run a real-time customer ↔ store conversation end to end."
---

This flow covers the two surfaces that keep a shopper engaged after they buy: **product reviews**
(write a review, list the approved ones, vote them helpful) and **messaging** (open a support
thread, exchange messages, and stream replies live). Both act on a **signed-in shopper's own
data**, so every call here pairs an API key with the shopper's session token in `xAuthToken`. (See
[Workflow Examples](/workflows/introduction) for the client setup, and
[Storefront Checkout](/workflows/storefront-checkout#3-sign-the-shopper-in) for how to mint that
session token.)

<Info>
  Reviews and messaging writes are **browser-origin storefront writes**, so the **publishable**
  client is fine for them — the shopper's `xAuthToken` is what proves ownership. Only the merchant
  admin can moderate reviews or reply from the store side; those are not API actions.
</Info>

## Reviews

### 1. Submit a review

A logged-in shopper submits a review for a product. It is created `status: "pending"` and only
appears in public listings once the merchant approves it in their admin. Pass an `order_id` that
belongs to the shopper to have the review flagged `verified_purchase: true`.

```typescript
const { review } = await storefront.reviews.submitReview({
  xAuthToken: session.access_token,
  requestBody: {
    product_id: '0186665f-f0a1-4c79-976a-919b5b1c2daa',
    rating: 5,
    title: 'Outstanding quality',
    body: 'Exactly as described — fast shipping and great build quality. Would buy again.',
  },
});
```

```json Response

    "id": "11bc03d9-d1cb-4833-b9d2-f0b016104cde",
```

<Note>
  A shopper may have only **one** non-rejected review per product — a second submission returns
  `409 Conflict`. The review stays hidden from `listReviews` (below) on a publishable key until the
  merchant approves it.
</Note>

### 2. List approved reviews

Render the review section of a product page. On a **publishable** key this returns only `approved`
reviews and always includes a `summary` block with the average rating and star distribution. No
customer token needed — this is public catalog data.

```typescript
const { reviews, summary } = await storefront.reviews.listReviews({
  productId: '0186665f-f0a1-4c79-976a-919b5b1c2daa',
});
```

</Tip>

### 3. Mark a review helpful

Let shoppers upvote the reviews they found useful so you can surface them first. Only approved
reviews accept votes; there's no built-in dedup, so track per-shopper voting in your own UI if you
need it.

```typescript
const { helpful_count } = await storefront.reviews.markReviewHelpful({
  id: '11bc03d9-d1cb-4833-b9d2-f0b016104cde',
});
// → helpful_count === 1
```

```json Response
{ "helpful_count": 1 }
```

<Note>
  A shopper can remove their own review with
  `storefront.reviews.deleteReview({ id, xAuthToken: session.access_token })` — it returns
  `{ "success": true, "message": "Review deleted" }`. Deleting someone else's review returns `403`.
</Note>

## Messaging

### 4. Start a conversation

Open a thread with the store and post the shopper's first message in one call. `thread_type`
classifies the inquiry (`general`, `order_inquiry`, `product_inquiry`, `support`, `complaint`,
`delivery`, `return`, `refund`, `technical`). When you pass `customer_id`, include the matching
`xAuthToken`.

```typescript
const { thread } = await storefront.messaging.createConversation({
  xAuthToken: session.access_token,
  requestBody: {
    customer_id: '88ad3683-d291-4c70-8f8b-9903a42bde1b',
    customer_name: 'Jane Doe',
    customer_email: 'jane.doe@example.com',
    subject: 'Question about my order',
    message: 'Hi — I placed an order and wanted to check the expected delivery window.',
    thread_type: 'order_inquiry',
  },
});
// → thread.id is the conversation you send follow-ups to
```

```json Response
{
  "thread": {
    "id": "06084af6-7334-43d6-82c0-935f92335a25",
    "subject": "Question about my order",
    "status": "active",
```

### 5. Send a message

Append the shopper's next message to the open thread. The token must own the thread, otherwise the
call returns `403`. (The message text goes in `message`.)

```typescript
const { message } = await storefront.messaging.sendMessage({
  xAuthToken: session.access_token,
  id: '06084af6-7334-43d6-82c0-935f92335a25',
  requestBody: { message: 'Thanks! Also, can I add gift wrapping to this order?' },
});
```

```json Response
{
  "message": {
    "id": "33e2c744-37a9-42df-a9df-447715526a41",
    "created_at": "2026-06-20T11:29:02.467216+00:00"
  }
}
```

### 7. Mark the thread read

When the shopper opens a conversation, clear their unread state.

```typescript
const result = await storefront.messaging.markThreadRead({
  xAuthToken: session.access_token,
  id: '06084af6-7334-43d6-82c0-935f92335a25',
});
```

```json Response
{
  "success": true,
  "thread_id": "06084af6-7334-43d6-82c0-935f92335a25"
}
```

### 8. Stream replies in real time

Polling for new messages is wasteful. Instead, request **realtime connection details** for the
thread, then hand them to the `subscribeToThread` helper to receive the store's replies the moment
they're sent — no polling loop.

```typescript
const conn = await storefront.messaging.getMessagingRealtimeToken({
  xAuthToken: session.access_token,
  requestBody: { thread_id: '06084af6-7334-43d6-82c0-935f92335a25' },
});
```

```json Response
{
  "mode": "direct",
  "realtime_url": "wss://realtime.tybritelabs.com",
  "realtime_key": "rt_pub_8f3c1a9e4b2d",
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.<redacted>.signature",
  "channel": "thread:06084af6-7334-43d6-82c0-935f92335a25",
  "thread_id": "06084af6-7334-43d6-82c0-935f92335a25",
  "environment": "production"
}
```

The fields are neutral connection details — `mode`, `realtime_url`, `realtime_key`, `channel`, and
(for `direct` mode) a short-lived `token`. Pass them straight into the SDK helper:

```typescript
import { subscribeToThread } from '@tybrite-labs/sdk';

const unsubscribe = await subscribeToThread({
  mode: conn.mode,                 // 'direct' | 'broadcast'
  realtimeUrl: conn.realtime_url,
  realtimeKey: conn.realtime_key,
  channel: conn.channel,
  token: conn.token,               // required for 'direct', ignored for 'broadcast'
  onMessage: (msg) => {
    console.log('New message:', msg.message_content);
  },
});

// later, when the shopper closes the thread:
unsubscribe();
```

<Note>
  The `mode` tells you how the shopper was identified:
  - **`direct`** — a shopper signed in through Galactic Core (`xAuthToken`). The `token` authorizes
    a direct subscription to **their own** thread.
  - **`broadcast`** — a shopper authenticated with your own identity provider (`xExternalAuth`),
    server-side integrations, and marketplace operators. Subscribe to the named `channel`; no
    `token` is used.

  The helper handles both — you just forward the fields. Want to bring your own identity provider?
  See [Customer authentication](/authentication#customer-sessions).
</Note>
