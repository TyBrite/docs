Galactic Core gives you two ways to manage who your customers are. Pick the one that matches your stack.

## How the two services split the work

Before we dig into the paths, the mental model:

- **`AuthenticationService` (`client.auth`)** owns **identity and sessions**: register, log in, log out, password reset, OTP, magic links, session refresh, and a tiny "who am I?" check (`getCurrentUser` → returns the user id and a few identifying fields). That's it.
- **`CustomersService` (`client.customers`)** owns **the customer profile resource**: provision a customer record, read the full profile (with addresses, store metrics, purchase history, status), update profile fields, and link to an external identity via `external_id`.

The split is not "Path A uses one service and Path B uses the other." Both services are part of both paths. Even a Path A storefront — where the customer logs in via Galactic Core — fetches and edits the customer's profile through `CustomersService`, not through `AuthenticationService`. The auth service tells you *who* the customer is; the customers service tells you *what* the customer looks like.

Where Path A and Path B differ is **how you prove the caller is the customer** when calling `CustomersService`:

| Path | What proves "I am this customer" | Header |
|---|---|---|
| A — GC-native auth | A Galactic Core session JWT (issued by `client.authentication.login`) | `xAuthToken` |
| B — bring your own auth | An HMAC-signed assertion from your backend, scoped to the customer's `external_id` | `xExternalAuth` |

Either header authorises the same operation. The customer-self endpoints (`getCustomer`, `updateCustomer`) accept both — the dev picks whichever matches their integration.

## The two paths

### Path A — Galactic Core handles auth

Use [`AuthenticationService`](./api-reference/classes/AuthenticationService) (`client.auth`) for registration, login, password resets, OTP, and session refresh. Each successful registration also creates the customer record automatically — you don't call `CustomersService` for creates.

**Best when:**
- You're building a storefront from scratch and want the fastest path to a working "create account / sign in" UX.
- You don't already have an identity provider.
- You're happy with email/password + OTP + magic links as the auth surface.

### Path B — Bring your own auth provider

Authenticate users in your own identity system (Auth0, Clerk, Cognito, Firebase Auth, NextAuth, your own SSO) and use [`CustomersService`](./api-reference/classes/CustomersService) (`client.customers`) to provision the matching Galactic Core customer record from your backend.

**Best when:**
- You already use Auth0/Clerk/Cognito/Firebase/NextAuth/SSO and don't want a second login system.
- You need MFA flows, social logins, enterprise SSO, or other features your provider already does well.
- Your buyers exist in an upstream system (ERP, CRM, accounting platform) and are projected into Galactic Core via integration.

## Decision matrix

| Question | Use Path A | Use Path B |
|---|:---:|:---:|
| Do you already have an identity provider? | No | Yes |
| Do you want Galactic Core to mint customer JWTs? | Yes | No |
| Do you want to handle MFA / social / SSO yourself? | Galactic Core does it | Yes |
| Are buyers created upstream (CRM, ERP, etc)? | Probably no | Probably yes |
| Is your storefront browser-only with no backend? | Yes (use publishable key + customer JWT) | Hard — you need a backend to call `CustomersService` |

You can mix both if you need to (e.g. start with Path A and migrate to Path B later). The customer record is the same row either way.

## How Path B works end-to-end

```
1. User signs up in your identity provider (Auth0, Clerk, ...)
   ↓
2. Your provider fires a post-signup webhook
   ↓
3. Your backend receives the webhook
   ↓
4. Backend calls POST /v1/customers with the user's profile + their upstream id
       client.customers.createCustomer({
         requestBody: {
           email, name, phone,
           external_id: 'auth0|66a3f8c2b1d9c204a1f7e3d1',
         }
       })
   ↓
5. Galactic Core returns { customer } — store customer.id alongside the
   upstream user record (or rely on external_id for lookups)
   ↓
6. For every subsequent order / cart / review, your backend passes
   customer_id to the relevant Galactic Core endpoint
```

The Galactic Core customer never has a Galactic Core JWT — they have your provider's session, and your backend translates between the two.

## `external_id` — the join key

`external_id` is an optional string on every customer record. It's **unique per store + environment**, which means you can rely on `(store_id, environment, external_id)` as a stable lookup key for the lifetime of the customer.

```typescript
// On signup webhook from Auth0
await client.customers.createCustomer({
  requestBody: {
    email: user.email,
    name: user.name,
    external_id: user.sub, // Auth0 subject — store it
  }
});

// Later, in any backend code that needs the Galactic Core customer:
// 1. You stored the GC customer.id when you created it → use that directly.
// 2. Or, store only the external_id and look up the GC customer when needed
//    via your own DB / cache, or by fetching `/v1/customers?external_id=…`
//    once that filter ships.
```

Set it on `createCustomer`, update it on `updateCustomer` if the upstream identifier changes (e.g. provider migration).

## Customer-self endpoints in Path B — the `x-external-auth` assertion

`GET /v1/customers/{id}` and `PATCH /v1/customers/{id}` are customer-self operations: the caller must prove they ARE that customer. In Path A this is a Galactic Core JWT (`x-auth-token`). In Path B the customer has no Galactic Core session — they have **your** session — so we accept an equivalent: an HMAC-signed assertion from your backend.

**Wire format:**

```
X-External-Auth: <base64url(JSON)>.<base64url(HMAC-SHA256(JSON))>
```

**The JSON claim:**

```json
{
  "external_id": "auth0|66a3f8c2b1d9c204a1f7e3d1",
  "iat": 1780045000,
  "exp": 1780045300
}
```

**Rules:**

- Signed with your store's `hmac_secret` (the same secret used for orders/payments signing — find it under Integrations → Developer).
- `exp - iat` must be **≤ 300 seconds**.
- The `external_id` in the claim must resolve to a Galactic Core customer whose `id` matches the `{id}` in the path — otherwise `403`.
- Provide `x-external-auth` **OR** `x-auth-token`, not both. Both → `400`.

**Signing example (Node.js):**

```typescript
import { createHmac } from 'node:crypto';

function signCustomerAssertion(externalId: string, hmacSecret: string) {
  const now = Math.floor(Date.now() / 1000);
  const claim = JSON.stringify({
    external_id: externalId,
    iat: now,
    exp: now + 240, // 4 minutes — well inside the 300s cap
  });
  const b64url = (b: Buffer) => b.toString('base64url');
  const sig = createHmac('sha256', hmacSecret).update(claim).digest();
  return `${b64url(Buffer.from(claim))}.${b64url(sig)}`;
}

// On a request from your storefront:
async function handleCustomerProfileFetch(req, res) {
  // 1. Verify the customer's session in YOUR identity provider
  const upstreamUser = await yourProvider.verifySession(req);
  if (!upstreamUser) return res.status(401).end();

  // 2. Sign the assertion for Galactic Core
  const assertion = signCustomerAssertion(upstreamUser.id, process.env.GC_HMAC_SECRET!);

  // 3. Call Galactic Core on the customer's behalf
  const r = await fetch(`https://api.tybritelabs.com/v1/customers/${gcCustomerId}`, {
    headers: {
      'Authorization': `Bearer ${process.env.GC_SECRET_KEY}`,
      'x-external-auth': assertion,
    },
  });
  return res.json(await r.json());
}
```

The dev's backend signs the assertion server-side; the storefront never sees `hmac_secret`. Each assertion is single-use (just expire fast), so a leaked assertion is useless after a few minutes.

## What Path B still doesn't give you

- **No password reset / magic link / OTP delivery.** Those live in your identity provider.
- **Sessions are your provider's job.** Galactic Core never issues a JWT to the customer; your provider does. The `x-external-auth` assertion is a per-request claim from your backend, not a long-lived customer token.

## What Path A doesn't give you

- **No external identity reuse.** Galactic Core mints its own JWTs; it doesn't accept tokens issued by Auth0/Clerk/etc.
- **No social logins or enterprise SSO** out of the box.

## Summary

Pick Path A if you're starting fresh and want a fast working storefront. Pick Path B if you already have an identity provider you're not going to throw away. The Galactic Core commerce primitives (orders, cart, products, reviews, recommendations) work identically either way — only the customer-record provisioning differs.
