The `MarketplaceService` class (accessed via `client.marketplace`) is the entry point for building a **multi-merchant marketplace storefront** — one shopfront where many independent merchants are aggregated into a single shopping experience, a single cart can span several merchants, and one payment is automatically split to each merchant.

If you're building a normal single-store storefront, you don't need this service at all — use `products`, `cart`, `orders`, etc. directly. This page is for marketplace deployments.

---

## Before you start: how a marketplace works in Tybrite

A few concepts make everything else click.

- **Marketplace vs. merchant.** A *marketplace* aggregates many *merchants* (each is an independent store with its own catalog, pricing, and inventory). The marketplace operator sets commission and branding; merchants keep full control of their own products.
- **The operator key.** A marketplace storefront authenticates with a single **marketplace operator key** (a publishable key, `tybrite_pk_...`, minted by the operator). You use this one key for everything the storefront does: browsing the aggregated catalog, rendering a single merchant's shop page, signing shoppers in, previewing carts, and checking out. You do **not** juggle per-merchant keys.
- **Aggregated reads.** When you call the normal catalog endpoints (`products`, `taxonomy`, `search`) with the operator key, they automatically return results across **every active merchant**, and each product/result is stamped with the `merchant_store_id` it came from. Pass `storeId` to narrow any of them to one merchant.
- **One shopper, every merchant.** A shopper signs up once and can buy from any merchant. Their identity is marketplace-wide; per-merchant relationships are created automatically the first time they buy from a merchant. `getMarketplaceCustomer` returns the unified picture.
- **Unified split checkout.** A single cart may contain items from several merchants. One `marketplaceCheckout` call creates one payment for the whole basket; after the shopper pays, each merchant is paid their share (gross minus commission) automatically. You never move money yourself.

<Note>
  **Use the marketplace operator key** for everything on this page (and for aggregated `products`/`taxonomy`/`search` reads). It's a publishable key, safe for the browser. Merchant-side setup (a merchant connecting their payout account) and operator back-office (payout reports) are **not** part of this API — they happen in the admin apps.
</Note>

## The typical integration flow

A marketplace storefront usually wires up in this order:

1. **Identify the marketplace** — `getMarketplaceInfo()` to get the name, logo, colors, and which capabilities are on.
2. **Browse** — `client.products.listProducts()`, `client.taxonomy.listCategories()`, `client.search.searchProducts()` with the operator key for the aggregated catalog.
3. **Shop a single merchant (optional)** — pass `storeId` to the reads above + `getMarketplaceInfo({ storeId })` for that merchant's shop page.
4. **Sign the shopper in** — `client.authentication.register` / `login` (operator key) → you get a shopper session token.
5. **Build a cart client-side** — collect line items, each tagged with its `merchant_store_id`.
6. **Check out** — `marketplaceCheckout()` → returns a `client_secret`; complete the payment with Stripe.js on the storefront.
7. **Order confirmation / account** — the order finalizes automatically on payment success; `getMarketplaceCustomer()` shows the shopper's cross-merchant history.

## Methods

### `getMarketplaceInfo`

Read information about the marketplace. This is usually the **first call** your storefront makes. It has **two modes**, controlled by whether you pass `storeId`:

#### Mode 1 — Marketplace identity & branding (no `storeId`)

Returns the marketplace's own name, branding, contact details, and which capabilities are enabled. Use it to render your storefront chrome (logo, colors, name) and to feature-detect (e.g. only show the fee line if `commission_enabled`).

```typescript
const marketplace = await client.marketplace.getMarketplaceInfo({});

// {
//   marketplace: true,
//   mode: "marketplace",
//   name: "Acme Marketplace",
//   logo_url: "https://.../logo.png",
//   primary_color: "#3B82F6",
//   tagline: "Everything, from everyone",
//   support_email: "help@acme.market",
//   description: "...",
//   website: "https://acme.market",
//   phone: "+1 555 0100",
//   address: "...",
//   unified_checkout: true,      // is cross-merchant checkout enabled?
//   commission_enabled: true     // does the marketplace charge commission?
// }

document.title = marketplace.name;
applyTheme(marketplace.primary_color);
```

#### Mode 2 — One merchant's full store context (`storeId`)

Returns the **same comprehensive payload** as `client.system.getStoreInfo`, but for a single merchant inside the marketplace. This is what you render on a merchant's **shop page header** — their business info, a catalog summary with a small sample of products, payment methods, shipping, CMS content, and feature flags — in one call. It's also ideal as context for an AI shopping agent describing one merchant.

```typescript
const merchant = await client.marketplace.getMarketplaceInfo({
  storeId: 'merchant-a-uuid',
  sections: 'catalog,shipping,features',   // optional: request only what you need
});

console.log(merchant.store.name, merchant.store.logo_url, merchant.store.description);
console.log(`${merchant.catalog?.products.total} products`);     // a COUNT, not the full list
console.log(merchant.catalog?.products.sample);                  // a thin sample (≤10) for context
```

<Tip>
  The catalog block returns **counts plus a small sample** of products (each with a few fields: id, name, category, price, image). It is intentionally **not** the full catalog — use `client.products.listProducts({ storeId })` to actually browse and paginate a merchant's products.
</Tip>

**Parameters**

| Param | Type | Notes |
| :--- | :--- | :--- |
| `storeId` | `string?` | Optional. A merchant's id. Omit for marketplace identity; supply to get one merchant's store context. The merchant must be active in the marketplace. |
| `sections` | `string?` | Optional, only with `storeId`. Comma-separated subset of `catalog,pricing,promotions,payments,shipping,cms,features`. Omit to return all. |

**Responses**

| Code | Meaning |
| :--- | :--- |
| `200` | Marketplace identity, or the merchant's store context. |
| `400` | `storeId` is not a valid UUID. |
| `403` | Called with a non-operator (single-store) key — use `client.system.getStoreInfo` instead. |
| `404` | No **active** merchant matches `storeId` (suspended/pending/unknown). |

### `marketplaceCheckout`

Check out a cart that may contain items from **multiple merchants** in one call. Each line item names the merchant it belongs to. Tybrite validates the items against the live marketplace catalog, resolves commission per merchant, and creates **one payment** for the entire basket.

The response gives you a `client_secret` to complete the payment on the storefront (with Stripe.js) and a per-merchant `merchant_breakdown` (gross, discount, commission, net) you can display.

You can also apply discounts. Pass an optional `discounts` array to apply a **merchant's own promotion or gift card** to that merchant's portion of the basket — each entry names a `merchant_store_id` and an optional `promotion_id` and/or `gift_card_code`, and the discount reduces only that merchant's subtotal. **Marketplace-wide promotions run by the operator apply automatically** — you don't pass them. The response then reports `discount_total`, the `operator_funded_discount` (the share the operator funded), a per-merchant `discount_breakdown`, and `discount_amount` + `merchant_gross` on each `merchant_breakdown` entry.

```typescript
const order = await client.marketplace.marketplaceCheckout({
  requestBody: {
    items: [
      { product_id: 'p1', variant_id: 'v1', merchant_store_id: 'merchant-a-uuid', quantity: 1 },
      { product_id: 'p2', variant_id: 'v2', merchant_store_id: 'merchant-b-uuid', quantity: 2 },
    ],
    customer_email: 'john.doe@example.com',
    customer_name: 'John Doe',
    currency: 'USD',
    shipping_address: { line1: '1 Market St', city: 'San Francisco', country: 'US' },
    billing_address:  { line1: '1 Market St', city: 'San Francisco', country: 'US' },
    // Optional: apply a merchant's own promotion / gift card to that merchant's portion.
    discounts: [
      { merchant_store_id: 'merchant-a-uuid', promotion_id: 'promo-uuid', gift_card_code: 'GC-XXXX-YYYY' },
    ],
  },
});

// Show the shopper what each merchant receives (already computed for you):
order.merchant_breakdown.forEach(m =>
  console.log(`${m.merchant_store_id}: gross ${m.gross_amount}, discount ${m.discount_amount}, fee ${m.commission_amount}, net ${m.net_amount}`)
);

// Total savings on the basket (per-merchant discounts + any marketplace-wide promotion):
console.log(`saved ${order.discount_total} (${order.operator_funded_discount} funded by the marketplace)`);

// Complete the payment on the storefront with the returned client_secret:
const stripe = Stripe(STRIPE_PUBLISHABLE_KEY);
const { error } = await stripe.confirmPayment({
  clientSecret: order.client_secret,
  confirmParams: { return_url: 'https://shop.example.com/order/confirmation' },
});
```

**Once the payment succeeds, the order finalizes automatically** — Tybrite creates each merchant's order, splits the funds, reduces stock, and links the shopper to each merchant. You do **not** make any follow-up API call.

<Note>
  **Stock is reserved at checkout.** When you call `marketplaceCheckout`, the items are **held** against each merchant's inventory immediately (so two shoppers can't buy the last unit at once), and the hold is converted to a real stock reduction when payment succeeds. If an item can't be held, the call fails with **`400 insufficient_stock`** and no order or payment is created. Holds expire automatically (a short window) if the payment is never completed, returning the stock to availability — so an abandoned checkout never permanently locks inventory.
</Note>

<Note>
  **You don't resolve commission separately.** The `merchant_breakdown` in this response already contains the per-merchant gross/commission/net. (If you need to show fees on the cart page *before* checkout, compute them from your own displayed prices, or call checkout when the shopper is ready — there is no separate public commission endpoint.)
</Note>

**Request body**

| Field | Type | Notes |
| :--- | :--- | :--- |
| `items` | array | Required. Each item: `{ product_id?, variant_id, merchant_store_id, quantity }`. `variant_id` and `merchant_store_id` are required per item. |
| `customer_email` | `string` | Required. The shopper's email. |
| `customer_name` | `string?` | Optional. |
| `customer_phone` | `string?` | Optional. |
| `currency` | `string?` | Optional ISO code; defaults to `USD`. |
| `shipping_address` | `object?` | Optional address object. |
| `billing_address` | `object?` | Optional address object. |
| `discounts` | `array?` | Optional per-merchant discounts. Each entry: `{ merchant_store_id, promotion_id?, gift_card_code? }`. Applies a merchant's own promotion/gift card to that merchant's portion of the basket. Marketplace-wide operator promotions apply automatically and are **not** passed here. |

**Response (`201`)**

```typescript
{
  order_group_id: string,       // the unified order's id
  group_number: string,         // human-friendly reference, e.g. "MKT-2026-000123"
  payment_intent_id: string,    // the payment intent (pi_...)
  client_secret: string,        // pass to Stripe.js to collect payment
  currency: string,
  subtotal: number,
  total_amount: number,
  discount_total?: number,             // total discount across the basket
  operator_funded_discount?: number,   // the share funded by the marketplace operator
  discount_breakdown?: Array<{
    merchant_store_id: string,
    merchant_discount: number,         // funded by this merchant's own promotion
    gift_card_amount: number,          // redeemed from a gift card against this merchant
    operator_funded_discount: number,  // operator-funded portion for this merchant
  }>,
  merchant_breakdown: Array<{
    merchant_store_id: string,
    merchant_gross: number,            // before discount
    discount_amount: number,           // discount applied to this merchant's portion
    gross_amount: number,              // after discount, before commission
    commission_amount: number,
    net_amount: number,
    commission_rule_id: string | null,
  }>,
}
```

**Responses**

| Code | Meaning |
| :--- | :--- |
| `201` | Unified order created; awaiting payment. |
| `400` | Invalid items/body (e.g. bad `variant_id`/`merchant_store_id`, missing email). |
| `403` | Not a marketplace operator key, or unified checkout is disabled for this marketplace. |
| `404` | An item isn't available on the marketplace (inactive merchant/product). |
| `500` / `502` | Server error or the payment provider could not be reached. |

### `getMarketplaceCustomer`

Return a shopper's **unified marketplace profile** — their identity, the merchants they've shopped with, and their cross-merchant order history — in one response. Use it for the shopper's account page.

This requires the shopper's session token (from sign-in) in the `X-Customer-Token` header, in addition to the operator key.

```typescript
// 1. Sign the shopper in (operator key) to get a session token:
const { session } = await client.authentication.login({
  requestBody: { email: 'john.doe@example.com', password: '...' },
});

// 2. Fetch their unified profile:
const profile = await client.marketplace.getMarketplaceCustomer({
  xCustomerToken: session.access_token,
});

console.log(profile.customer.email);
console.log(`${profile.stores.length} merchant relationships`);
console.log(`${profile.orders.length} orders across all merchants`);
```

**Response (`200`)**

```typescript
{
  customer: { ids: string[], email, name, phone, ... },  // unified identity
  stores:   Array<{ store_id, ...affiliation }>,          // per-merchant relationships
  orders:   Array<{ ... }>,                               // cross-merchant order history
}
```

**Responses**

| Code | Meaning |
| :--- | :--- |
| `200` | The unified profile. |
| `401` | Missing/invalid operator key or customer token. |
| `403` | Not a marketplace operator key. |

## Aggregated reads (the marketplace catalog)

You don't use `MarketplaceService` for browsing — you use the normal services with the **operator key**, and they automatically aggregate across every active merchant. Each product/result carries its source `merchant_store_id`.

```typescript
const market = new Tybrite({ apiKey: 'tybrite_pk_live_operator_...' });

// Combined catalog across all merchants (each product has merchant_store_id):
const { products } = await market.products.listProducts({ limit: 24 });

// Deduplicated taxonomy across merchants:
const { categories } = await market.taxonomy.listCategories();

// Cross-merchant search:
const results = await market.search.searchProducts({ q: 'headphones' });
```

### Single merchant "shop page"

To render one merchant's storefront *inside* the marketplace, pass `storeId` (the merchant's id) to the aggregated reads. Everything narrows to that one (active) merchant:

```typescript
const storeId = 'merchant-a-uuid';

const header        = await market.marketplace.getMarketplaceInfo({ storeId });   // shop-page header
const { products }  = await market.products.listProducts({ storeId });            // that merchant's catalog
const { categories} = await market.taxonomy.listCategories({ storeId });          // that merchant's categories
const results       = await market.search.searchProducts({ q: 'case', storeId }); // search within the merchant
```

### Reading a product's reviews on the marketplace

With the operator key, you can **read approved reviews** for a product, but you must say which merchant owns it via `storeId` (required for operator keys). Submitting, deleting, or voting on reviews is still done with a shopper session, not the operator key.

```typescript
const reviews = await market.reviews.listReviews({
  productId: 'product-uuid',
  storeId:   'merchant-a-uuid',   // required when using an operator key
});
```

<Note>
  `storeId` only narrows results when you use an **operator** key. With a single-store key it's ignored (you're already scoped to one store). On `reviews`, an operator key **requires** `storeId` and may only read approved reviews.
</Note>

### Returns on the marketplace

A return always belongs to **one merchant** — the merchant whose order is being returned. So with the operator key, every `client.returns` shopper action takes a required `storeId` (the merchant from the order) alongside the shopper's session. A shopper's returns are therefore **per-merchant**: to show "all my returns" across the marketplace, fetch each merchant the shopper ordered from and merge the lists in your storefront.

```typescript
const storeId = 'merchant-a-uuid';   // the merchant whose order this is

// lodge a return for the signed-in shopper at that merchant
const { data: lodged } = await market.returns.createReturn({
  storeId, xAuthToken: shopperSession,
  requestBody: { order_id, reason_code: 'damaged', items },
});

// the shopper's returns + store-credit at that merchant
const { data: mine }    = await market.returns.listReturns({ storeId, xAuthToken: shopperSession });
const { data: balance } = await market.returns.getStoreCredit({ storeId, xAuthToken: shopperSession });
```

The same `storeId` applies to `getReturn`, `acceptReturnCredit`, and `requestReturnRefund`. The `storeId` must be an active merchant. See the [Returns](/sdk/api-reference/classes/ReturnsService) reference for the full method set and the store-credit flow.

<Note>
  This is a shopper acting on their **own** returns at one merchant. There is no cross-merchant "all returns" feed or store-credit-owed summary in the API — operator oversight of returns is a back-office concern, handled in the operator's admin, not through the SDK.
</Note>

## Curated collections (homepage merchandising)

A marketplace operator can curate **merchandising sections** for the storefront — for example "Trending now", "Featured shops", or a seasonal promotion rail. Each section groups products, merchants, or promotions the operator hand-picked (or auto-curated) to merchandise the storefront. Use them to build your homepage.

Two operator-key methods cover this: `listMarketplaceCollections` returns the sections, and `getMarketplaceCollection` returns one section's resolved members. Curating the sections themselves is an operator back-office task in the Super Admin app — not part of this API.

### `listMarketplaceCollections` — the storefront's curated sections

```typescript
const market = new Tybrite({ apiKey: 'tybrite_pk_live_operator_...' });

const { collections } = await market.marketplace.listMarketplaceCollections({
  placementKind: 'product',   // optional: 'product' | 'merchant' | 'promotion'
});

// Render each section in display order:
collections
  .sort((a, b) => (a.display_priority ?? 0) - (b.display_priority ?? 0))
  .filter((c) => c.show_on_homepage)
  .forEach((c) => {
    // c.slug, c.name, c.description, c.placement_kind
    renderSectionHeader(c.name, c.description);
  });
```

```json Response
{
  "collections": [
    {
      "slug": "trending-now",
      "name": "Trending now",
      "description": "The most popular products across the marketplace this week.",
      "placement_kind": "product",
      "display_priority": 10,
      "show_on_homepage": true
    },
    {
      "slug": "featured-shops",
      "name": "Featured shops",
      "description": "Merchants worth a look.",
      "placement_kind": "merchant",
      "display_priority": 20,
      "show_on_homepage": true
    }
  ]
}
```

| Param | Required | Notes |
|---|---|---|
| `placementKind` | no | Return only sections of this kind: `product`, `merchant`, or `promotion`. |

### Blending curated collections with sponsored + organic

Curated collections, sponsored placements, and organic recommendations are complementary — a typical homepage interleaves all three. Fetch them together:

```typescript
const [{ collections }, { placements }, recs] = await Promise.all([
  market.marketplace.listMarketplaceCollections({}),
  market.marketplace.getAdSlot({ slotKey: 'home_hero', limit: 3 }),
  market.recommendations.getRecommendations({ type: 'trending', limit: 12 }),
]);

// Resolve the members of each curated section, then lay out the homepage:
const sections = await Promise.all(
  collections.map((c) => market.marketplace.getMarketplaceCollection({ slug: c.slug, limit: 12 })),
);

// e.g. hero (sponsored, labelled) → curated sections → organic "you might like" rail.
```

## Advertising / sponsored placements

A marketplace storefront can show **placements** in named **slots** (a hero banner, a category strip, a search-results rail), blended in with your organic listings or recommendations. A slot returns two kinds of placement:

- **Sponsored** (`sponsored: true`) — paid spots that promote a merchant's products or shop. These **must** be rendered with their `disclosure_label`.
- **Curated fallback** (`curated: true, sponsored: false`) — placements the marketplace operator hand-picks to fill a slot when no paid placement is available. These are **not** labelled as sponsored.

Two operator-key methods cover the whole storefront side: `getAdSlot` fetches what to show in a slot, and `logAdEvent` reports impressions and clicks. Booking placements, setting rates, and managing slot inventory are **operator back-office** tasks done in the Super Admin app — they're not part of this API.

<Warning>
  Render the `disclosure_label` (for example `"Sponsored"`) on **every placement where `sponsored` is `true`** so shoppers can tell it apart from organic results. This is a requirement, not a suggestion. Curated fallback placements (`sponsored: false`) are not paid advertising and do not carry the label.
</Warning>

### `getAdSlot` — what to show in a slot

```typescript
const market = new Tybrite({ apiKey: 'tybrite_pk_live_operator_...' });

const { slot, context, placements } = await market.marketplace.getAdSlot({
  slotKey: 'home_hero',     // the named slot you're filling
  context: 'electronics',   // optional: a category id or search term for context-scoped slots
  limit: 5,                 // optional
});

for (const p of placements) {
  // p.booking_id        — pass this back when logging beacons for this placement
  // p.merchant_store_id  — the merchant being promoted
  // p.placement_kind     — 'product' | 'merchant' | 'promotion'
  // p.product_ids        — products to feature
  // p.promotion_id       — for 'promotion' placements, the promotion to feature (else null)
  // p.creative           — headline / image / cta to render (shape varies by slot)
  // p.sponsored          — true for paid ads, false for operator-curated fallback fill
  // p.curated            — true when the operator hand-picked this fallback placement
  // p.disclosure_label   — render this only when p.sponsored is true
  if (p.sponsored) renderSponsored(p, { label: p.disclosure_label });
  else renderCurated(p);   // curated fallback — no "Sponsored" label
}
```

```json Response
{
  "slot": "home_hero",
  "context": null,
  "placements": [
    {
      "booking_id": "7c9e6679-7425-40de-944b-e07fc1f90ae7",
      "merchant_store_id": "3f1a2b4c-1111-2222-3333-444455556666",
      "placement_kind": "product",
      "product_ids": [
        "715b8c10-eb11-4289-8c0e-2966e78cf9a5",
        "9cd6e472-de84-435c-be43-957b56682f43"
      ],
      "creative": {
        "headline": "Summer audio sale",
        "image_url": "https://cdn.example.com/ads/acme-hero.png",
        "cta": "Shop now"
      },
      "sponsored": true,
      "disclosure_label": "Sponsored"
    }
  ]
}
```

| Param | Required | Notes |
|---|---|---|
| `slotKey` | yes | The slot to fill, e.g. `home_hero`, `category_top`. |
| `context` | no | A category id or search term for context-scoped slots. |
| `limit` | no | Max placements to return. |

<Tip>
  For a `'promotion'` placement (or a promotion item in a curated collection), the
  payload gives you a `promotion_id` and a `merchant_store_id`. Fetch the promotion's
  detail — and the products it features — with
  `client.promotions.getPromotion({ id: promotion_id, storeId: merchant_store_id })`.
  See **PromotionsService → Getting the products in a promotion** for how each
  promotion `type` (percentage / fixed / bundle / bogo) resolves its products.
</Tip>

### Blending with organic recommendations

Sponsored placements sit *alongside* organic results, not in place of them. A common pattern is to fetch both and interleave:

```typescript
const [{ placements }, recs] = await Promise.all([
  market.marketplace.getAdSlot({ slotKey: 'home_hero', limit: 3 }),
  market.recommendations.getRecommendations({ type: 'trending', limit: 12 }),
]);

// Show sponsored placements first (labelled), then organic recommendations.
const feed = [
  ...placements.map((p) => ({ kind: 'sponsored', item: p })),
  ...recs.recommendations.map((r) => ({ kind: 'organic', item: r })),
];
```

### `logAdEvent` — impression & click beacons

Fire one **impression** per placement when it becomes visible, and a **click** when the shopper clicks it. Beacons are fire-and-forget.

```typescript
// When a placement scrolls into view:
await market.marketplace.logAdEvent({
  requestBody: {
    booking_id: placement.booking_id,
    event_type: 'impression',
    session_id: storefrontSessionId,   // optional, helps de-duplicate
  },
});
// → 202 { recorded: true, billable: true }

// When the shopper clicks it:
await market.marketplace.logAdEvent({
  requestBody: {
    booking_id: placement.booking_id,
    event_type: 'click',
    product_id: clickedProductId,      // optional
    session_id: storefrontSessionId,
  },
});
// → 202 { recorded: true, billable: true }
```

The response is `{ recorded: true, billable }`. `billable` tells you whether the event counted toward billing after de-duplication and invalid-traffic filtering — it's informational, so you don't need to act on it.

---

## What is NOT in this API

These are deliberately **not** SDK methods, because they're administrative, not storefront actions:

- **A merchant connecting their payout account** (Stripe Connect onboarding) — the merchant does this from their own admin/POS, not over the storefront API.
- **Running or viewing operator payout reports** — the operator does this from the Super Admin app.
- **Setting commission rules / marketplace branding** — configured by the operator in the Super Admin app; you only *read* the resulting branding via `getMarketplaceInfo`.

So the public marketplace surface is exactly: `getMarketplaceInfo`, `marketplaceCheckout`, `getMarketplaceCustomer`, `listMarketplaceCollections`, `getMarketplaceCollection`, `getAdSlot`, `logAdEvent`, plus the operator-key behavior of the catalog/search/auth/reviews/recommendations reads.

## Putting it together — a minimal storefront

```typescript
import { Tybrite } from '@tybrite-labs/sdk';

const market = new Tybrite({ apiKey: 'tybrite_pk_live_operator_...' });

// 1. Chrome
const info = await market.marketplace.getMarketplaceInfo({});
renderHeader(info.name, info.logo_url, info.primary_color);

// 2. Aggregated catalog (each product stamped with its merchant)
const { products } = await market.products.listProducts({ limit: 24 });

// 3. Shopper builds a cross-merchant cart in the browser:
const cart = [
  { product_id: 'p1', variant_id: 'v1', merchant_store_id: products[0].merchant_store_id, quantity: 1 },
  { product_id: 'p2', variant_id: 'v2', merchant_store_id: products[5].merchant_store_id, quantity: 2 },
];

// 4. Sign in (or register) the shopper:
const { session } = await market.authentication.login({
  requestBody: { email: 'john.doe@example.com', password: '...' },
});

// 5. Checkout → one payment, auto-split:
const order = await market.marketplace.marketplaceCheckout({
  requestBody: {
    items: cart,
    customer_email: 'john.doe@example.com',
    currency: 'USD',
  },
});

// 6. Collect payment with the returned client_secret (Stripe.js), then you're done —
//    the order finalizes and each merchant is paid automatically on success.
```

## Error handling

All marketplace methods follow the standard [response-code contract](/sdk/introduction#response-codes) and throw a typed `ApiError`:

```typescript
import { ApiError } from '@tybrite-labs/sdk';

try {
  await market.marketplace.marketplaceCheckout({ requestBody: { items, customer_email } });
} catch (err) {
  if (err instanceof ApiError) {
    if (err.status === 404) {/* an item is no longer available — refresh the cart */}
    if (err.status === 403) {/* not an operator key, or unified checkout disabled */}
    if (err.status === 502) {/* payment provider unavailable — ask the shopper to retry */}
  }
  throw err;
}
```
