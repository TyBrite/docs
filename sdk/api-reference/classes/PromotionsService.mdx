The `PromotionsService` class (accessed via `client.promotions`) handles active store discounts, marketing campaigns, and coupon validation logic. Promotions support an `image` field for displaying visual banners and campaign assets.

## Methods

### `listPromotions`

Retrieve a list of promotions based on their lifecycle status. This is useful for building a "Deals" or "Coupons" page on your storefront.

```typescript
const { promotions, pagination } = await client.promotions.listPromotions({
  status: 'active',
  cartTotal: '120.00',
  limit: 50,
  fields: 'id,name,type,value,min_purchase,image'
});

console.log(`Available promotions: ${promotions.length}`);
```

**Filter: `cartTotal`**
Providing a `cartTotal` allows the engine to pre-filter promotions that the customer is actually eligible for based on their current shopping bag value (compared against each promotion's `min_purchase`).

---

### `getPromotion`

Fetch full configuration data for a promotion, including its exact discount logic, validity dates, and the products it applies to.

```typescript
const promo = await client.promotions.getPromotion({
  id: 'promotion_uuid',
  fields: 'name,type,value,min_purchase,image' // Optional
});

if (promo.image) {
  console.log(`Promotion Banner: ${promo.image}`);
}
```

#### Getting the products in a promotion

The `type` field tells you how a promotion works and which product fields to read. The product fields hold product **ids** â€” fetch each product's full detail with `client.products.getProduct(...)` to render it.

| `type` | What to render | Products to resolve |
| :--- | :--- | :--- |
| `discount` (percentage) / `fixed` | A cart-level discount: show `value` and any `min_purchase`. | **None** â€” these promotions have no product list. |
| `bundle` | A bundle deal. | `bundle_products`: `[{ productId, quantity, discountedPrice }]` |
| `bogo` | Buy X, get Y free. | `bogo_required_products` (buy) + `bogo_free_products` (free): `[{ productId, quantity }]` |

```typescript
const promo = await client.promotions.getPromotion({ id: 'promotion_uuid' });

if (promo.type === 'bundle') {
  const items = await Promise.all(
    (promo.bundle_products ?? []).map(async (b) => ({
      ...b,
      product: await client.products.getProduct({ id: b.productId }),
    }))
  );
  // render each item.product with its b.discountedPrice
}
```

<Tip>
  **Marketplace storefronts.** When a sponsored ad placement or a curated collection
  hands you a `promotion_id`, fetch the promotion here by passing **`storeId`** â€” the
  `merchant_store_id` you received alongside it â€” and resolve its products against that
  same merchant:

  ```typescript
  const { merchant_store_id, promotion_id } = placement; // from getAdSlot / getMarketplaceCollection
  const promo = await client.promotions.getPromotion({ id: promotion_id, storeId: merchant_store_id });
  // for bundle/bogo, resolve products with client.products.getProduct({ id, storeId: merchant_store_id })
  ```

  A marketplace key may read a single promotion this way; it cannot list promotions.
</Tip>

## Data Schema

### `Promotion` Structure
| Field | Type | Description |
| :--- | :--- | :--- |
| `id` | `uuid` | Unique promotion identifier. |
| `name` | `string` | Promotion display name. |
| `type` | `enum` | `fixed`, `discount`, `bundle`, `bogo`. |
| `display_type` | `string` | Human-readable label derived from `type` (e.g. "Percentage Discount"). |
| `value` | `string` | The discount amount or percentage, depending on `type`. |
| `min_purchase` | `number` | Minimum order total for the promotion to apply. |
| `start_date` / `end_date` | `date` | Validity window. Only `active` promotions within this window apply. |
| `status` | `string` | Lifecycle status. |
| `image`| `string`| URL of the promotion banner. |
| `bundle_products` / `bogo_required_products` / `bogo_free_products` / `free_products` | `array` | Product sets for `bundle` / `bogo` promotions. |
| `has_time_restrictions` | `boolean` | Whether the promotion only applies on certain days/times (`start_time`, `end_time`, `time_zone`, `apply_to_days`). |

---

<Tip>
  **Pro Marketing Tip:** Use the `image` field to build high-impact promotional carousels. By associating visual assets directly with your discount logic, you can ensure that your marketing banners always reflect the current campaign status and validity.
</Tip>

## Response Codes

All promotion endpoints are `GET` and accept both **publishable** and **secret** keys. Marketplace keys may call `getPromotion` (with `storeId`) but not `listPromotions`. The `status` query parameter is validated against a strict enum.

| Code | Meaning |
| :--- | :--- |
| `200` | Success. |
| `400` | Invalid `status` enum (must be one of `active`, `inactive`, `scheduled`, `expired`), malformed query parameter, or a marketplace `getPromotion` call missing `store_id`. |
| `403` | A marketplace key tried to list promotions, or write. Use `getPromotion` with `store_id` to read a single promotion. |
| `401` | Invalid or missing API key. |
| `404` | Promotion not found (single-resource endpoint only). List endpoint returns `200` with an empty array. |
| `429` | Rate limit exceeded. |
| `500` | Internal server error. |
